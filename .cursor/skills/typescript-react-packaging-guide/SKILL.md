# Library Packaging & Deployment Guide

## Overview

Strategy for packaging React component libraries with Vite, deploying to GitLab Package Registry, and managing versions with Changesets in a multi-repo environment.

## Repository Strategy: Multi-Repo

- Library and service app (Next.js) reside in separate repositories
- Library has an independent lifecycle: versioning, CI/CD pipeline, release cadence
- Multiple service projects consume from a single central GitLab Package Registry

## Package Output Configuration

### package.json Fields

| Field | Format | Purpose |
|-------|--------|---------|
| `main` | CJS | Entry point for legacy environments (CommonJS) |
| `module` | ESM | Entry point for modern environments (ES Modules) |
| `types` | `.d.ts` | TypeScript type definitions for IDE support |
| `exports` | Sub-path map | Fine-grained module resolution for tree-shaking |

### Build Output

- ESM format: Optimized for Next.js and modern bundlers
- CJS format: Backward compatibility for older environments
- `.d.ts` files: Auto-generated by `vite-plugin-dts` for TypeScript consumers
- External dependencies: `react`, `react-dom` excluded via `peerDependencies`

## GitLab Package Registry

### .npmrc Configuration (Project Level)

- Both library (publisher) and service (consumer) projects need `.npmrc`
- Scope-based registry: Direct `@team-scope` packages to GitLab registry
- Public npm packages continue to resolve from default npm registry
- Authentication via deploy tokens or CI/CD variables

### Publishing

- Library CI pipeline runs `npm publish` on main branch merge
- Scoped package name: `@team-scope/package-name`
- Only packages matching the scope route to GitLab; all others use npm

## Version Management: Changesets

### Semantic Versioning (SemVer)

| Bump | Format | When |
|------|--------|------|
| **Major** | `X.0.0` | Breaking changes, API removal, structural overhaul |
| **Minor** | `0.X.0` | New components or features, backward-compatible |
| **Patch** | `0.0.X` | Bug fixes, performance improvements, no new features |

### Developer Workflow

1. Complete feature development
2. Run `npx changeset` to create a change description file
3. Select change type (major/minor/patch) and describe the change
4. Commit changeset file alongside code changes
5. Push and create Merge Request

### Release Workflow (Release Manager)

1. Pull latest `main` branch
2. Run `npx changeset version` to bump version and generate CHANGELOG
3. Commit with message `chore: release vX.Y.Z`
4. Push and trigger CI pipeline
5. Click manual `production_release` stage in GitLab CI

## Snapshot Releases (Pre-release)

### Purpose

- Test unreleased features in real service environments before official release
- Avoid polluting stable version line with potentially buggy code

### Process

- Run `changeset version --snapshot` to generate temporary version (e.g., `1.0.0-next.202310271530`)
- Publish with `npm publish --tag snapshot`
- Consumers install explicitly: `npm install @team-scope/lib@snapshot`
- CHANGELOG is not modified for snapshot releases

## Local Testing: yalc

### Why yalc Over npm link

- `npm link` creates symlinks that cause file path resolution issues
- yalc simulates a real local npm registry with proper package installation
- Matches actual deployment behavior more closely

### Workflow

1. **Library**: `yalc publish` - Stores built package in local yalc registry
2. **Service App**: `yalc add @team-scope/lib` - Installs from local registry
3. **Iterate**: `yalc push` - Pushes updates to all linked projects instantly
4. **Cleanup**: `yalc retreat` - Removes yalc links and restores original state

## CI/CD Pipeline Stages

| Stage | Trigger | Action |
|-------|---------|--------|
| **Build & Test** | PR/MR created | Run Vitest, Storybook build verification |
| **Snapshot** | PR/MR approved | Auto-publish snapshot version for review |
| **Release** | Main merge + manual trigger | `changeset version` → `npm publish` |
